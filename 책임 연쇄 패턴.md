# 책임 연쇄 패턴

(Chain of Responsibility, CoR)

> 💡 **“내가 안 되면 다음 사람에게 토스!”**
>
> 책임 연쇄 패턴은 **요청을 처리할 수 있는 여러 객체(핸들러)들을 사슬(체인)처럼 엮어두고, 요청이 들어오면 해결할 수 있는 객체를 만날 때까지 차례대로 넘기는 패턴**입니다.

## 📞 실생활 비유

새 부품을 컴퓨터에 설치했는데 리눅스 운영체제에서 작동하지 않아 고객센터에 전화를 걸었습니다.

> **1단계 (자동 응답기):** "안내해 드립니다. 1번은 윈도우, 2번은 맥..." (리눅스는 없으므로 **다음 단계로 토스**)
>
> **2단계 (일반 상담원):** "컴퓨터 껐다 켜보셨나요?" (내 문제를 해결 못 하므로 기술 부서로 **다음 단계로 토스**)
>
> **3단계 (리눅스 전문 엔지니어):** "아, 그 부품은 특정 드라이버를 깔아야 합니다!" (**해결! 체인 종료**)

이처럼 "내가 해결할 수 있으면 해결하고, 모르면 다음 단계로 넘긴다"는 흐름이 바로 책임 연쇄 패턴의 핵심입니다.

## 왜 필요할까?

### ❌ 패턴을 안 쓰면 발생하는 문제 (스파게티 코드)

온라인 쇼핑몰 주문 시스템을 만든다고 가정해 봅시다. 주문이 완료되려면 여러 검사를 거쳐야 합니다.

- IP 차단 검사 ➡️ 로그인 인증 검사 ➡️ 데이터 포맷 검증 ➡️ 캐시 데이터 확인 ➡️ 실제 주문 처리

처음엔 한 코드(함수) 안에 이 검사들을 쭉 적어 내려갑니다. 하지만 검사가 늘어날 때마다 거대한 `if-else` 지옥이 펼쳐지고, 코드가 너무 복잡해져서 수정하기도 무서워집니다.

```jsx
// 패턴 미사용: 거대하고 뚱뚱한 하나의 함수
function 처리_주문(request) {
    // 1. IP 차단 검사
    if (isBannedIP(request.ip)) {
        return "차단된 IP";
    }
    
    // 2. 로그인 인증 검사 (수정하려면 이 복잡한 함수 한가운데를 뜯어야 함!)
    if (!request.user.isLoggedIn) {
        return "로그인 필요";
    }
    
    // 3. 데이터 포맷 검증
    if (!isValidData(request.data)) {
        return "잘못된 데이터";
    }
    
    // 4. 캐시 확인 -> 5. 실제 주문 처리... (코드가 끝없이 길어짐)
    return "주문 완료";
}
```

**문제점:** 만약 "2번 로그인 인증" 방식을 바꾸고 싶거나, 중간에 "해외 IP 차단"이라는 1.5단계 검사를 끼워 넣으려면, **이 거대한 함수 한가운데를 비집고 들어가서 코드를 칼질**해야 합니다. 그러다 실수로 3, 4번 코드를 건드리면 전체 시스템이 고장 납니다.

### ⭕ 책임 연쇄 패턴을 썼을 때의 해결책

각각의 검증 단계를 독립된 '핸들러(객체)'로 만듭니다.

- `IP 검사 핸들러` ➡️ `인증 핸들러` ➡️ `데이터 유효성 핸들러`를 기차 기둥처럼 차례대로 연결합니다.
- **IP 검사 핸들러**가 "이상 없는 IP군!" 하면 다음 **인증 핸들러**로 요청을 넘깁니다.
- 만약 중간에 **인증 핸들러**에서 실패하면? 뒤의 검사들은 볼 것도 없이 **그 자리에서 요청을 중단**하고 끝냅니다.

```jsx
// 패턴 사용: 각자 자기 일만 하는 독립된 클래스들 (전문가들이 연결된 이어달리기)
class IpCheckHandler {
    handle(request) {
        if (isBannedIP(request.ip)) return "차단";
        return this.next.handle(request); // "난 통과! 다음 사람 토스~"
    }
}

class AuthHandler {
    handle(request) {
        if (!request.user.isLoggedIn) return "로그인 필요";
        return this.next.handle(request); // "난 통과! 다음 사람 토스~"
    }
}
```

### 🔑 결정적인 차이점

| **상황** | **❌ 패턴 안 쓴 기존 방식** | **⭕ 책임 연쇄 패턴** |
| --- | --- | --- |
| **중간에 검사 단계를 추가할 때**<br>(예: 매크로 방지 캡차 추가) | 거대한 주문 함수 내부를 직접 열어서 `if`문을 새로 코딩해야 함. (기존 코드 변형 위험 🚨) | `CaptchaHandler` 클래스를 새로 하나 만들어서 사슬 중간에 슥 끼워 연결하면 끝! |
| **순서를 바꾸고 싶을 때**<br>(예: 인증을 IP 검사보다 먼저) | `if`문들의 위치를 위아래로 옮기며 꼬이지 않게 코드를 다 재조립해야 함. | 코드는 건드릴 필요 없이, 조립하는 부분에서 `auth.setNext(ipCheck)` 로 순서만 바꿔주면 끝! |
| **다른 곳에서 재사용할 때**<br>(예: 자유게시판 글쓰기에도 IP 검사 필요) | 주문 함수에 있는 IP 검사 로직을 복사해서 게시판 함수에 또 붙여넣어야 함. | 이미 만들어진 `IpCheckHandler` 객체를 가져와서 게시판 사슬 맨 앞에 연결하면 끝! |

## ✨ 이 패턴의 장점과 단점

### 👍 장점

- **결합도가 낮아짐:** 요청을 보내는 사람(클라이언트)은 최종적으로 누가 이 요청을 처리할지 몰라도 됩니다. 그냥 첫 번째 핸들러에게 전달만 하면 됩니다.
- **단일 책임 원칙(SRP):** 검사 로직(인증, 캐시, 보안 등)을 각각 독립된 클래스로 쪼갤 수 있어 코드가 깨끗해집니다.
- **개방-폐쇄 원칙(OCP):** 기존 코드를 건드리지 않고, 새로운 핸들러(새로운 검사 단계)를 중간에 쉽게 끼워 넣거나 순서를 바꿀 수 있습니다.

### 👎 단점

- **누구도 처리하지 못할 수 있음:** 체인의 끝까지 갔는데도 아무도 요청을 처리하지 못하면 요청이 공중분해 될 수 있습니다. (예: "죄송합니다. 담당 부서가 없습니다" 하고 끊기는 전화)
- **디버깅이 조금 까다로움:** 요청이 여러 객체를 타고 전달되기 때문에, 문제가 생겼을 때 어디서 막혔는지 흐름을 추적하기 까다로울 수 있습니다.

### 🎯 요약하자면

사용자가 경험하는 실행 흐름은 동일합니다. 하지만 개발자 입장에서 기존 방식은 "한 몸으로 뭉쳐 있는 쇠사슬"이라 끊기가 어렵고, 책임 연쇄 패턴은 "언제든 뺐다 꼈다 할 수 있는 조립식 레고 체인"인 것입니다. 이 '유연함'과 '유지보수의 편리함' 때문에 이 패턴을 사용합니다!
주로 **웹 프레임워크의 필터(Filter), 미들웨어(Middleware)**, 혹은 GUI의 이벤트 전달 시스템에서 매우 흔하게 사용되는 강력한 패턴입니다.

---

## 🛠️ 패턴의 핵심 구조

1. **핸들러 인터페이스 (Handler):** 요청을 처리하는 메서드(`handle()`)와 다음 핸들러를 지정하는 메서드가 정의되어 있습니다. **모든 핸들러는 같은 인터페이스를 따라야 합니다.**
2. **기초 핸들러 (Base Handler):** 다음 핸들러가 누구인지 저장하는 필드와, 기본적으로 다음 핸들러로 요청을 넘기는 공통 코드가 들어있습니다.
3. **구상 핸들러 (Concrete Handlers):** 실제로 자기가 맡은 일(예: 인증, 캐싱 등)을 처리하는 진짜 객체들입니다.
4. **클라이언트 (Client):** 이 핸들러들을 `핸들러A ➡️ 핸들러B ➡️ 핸들러C` 형태로 동적으로 연결(체인 구성)하고 요청을 보냅니다.

---

## 실무 사례

책임 연쇄 패턴은 실무 코드에서 **"하나의 요청이 통과해야 하는 문(Gate)이 여러 개 있을 때"** 또는 **"이벤트가 발생했을 때 이를 처리할 적절한 주인을 찾을 때"** 가장 많이 쓰입니다.

### 1. 웹 서버의 미들웨어 & 필터 (가장 흔한 사례)

스프링(Spring)의 Security Filter Chain이나 노드(Node.js)의 Express 미들웨어가 대표적입니다. 사용자가 마이페이지를 조회할 때, `보안 인증 ➡️ 로깅 ➡️ 권한 확인` 단계를 차례대로 거치게 하는 구조입니다.

```jsx
// 인터페이스 역할을 할 기본 핸들러
class Middleware {
    setNext(nextMiddleware) {
        this.nextMiddleware = nextMiddleware;
        return nextMiddleware; // 체이닝을 위해 다음 객체 반환
    }

    handle(request) {
        if (this.nextMiddleware) {
            return this.nextMiddleware.handle(request);
        }
        return true; // 체인의 끝에 도달하면 최종 통과
    }
}

// 1단계: 인증 확인 핸들러
class AuthenticationMiddleware extends Middleware {
    handle(request) {
        if (!request.isLoggedIn) {
            console.log("❌ 인증 실패: 로그인이 필요합니다.");
            return false; // 여기서 체인을 끊어버림!
        }
        console.log("✅ 인증 성공");
        return super.handle(request); // 다음 핸들러로 토스
    }
}

// 2단계: 권한 확인 핸들러
class RoleCheckMiddleware extends Middleware {
    handle(request) {
        if (request.role !== 'ADMIN') {
            console.log("❌ 권한 실패: 관리자만 접근 가능합니다.");
            return false; // 여기서 체인을 끊어버림!
        }
        console.log("✅ 권한 확인 완료");
        return super.handle(request); // 다음 핸들러로 토스
    }
}

// --- 실무 사용 예시 ---
const auth = new AuthenticationMiddleware();
const role = new RoleCheckMiddleware();

// 체인 연결: 인증 ➡️ 권한 순서로 검사해라!
auth.setNext(role);

// 일반 유저의 요청이 들어왔다면?
const userRequest = { isLoggedIn: true, role: 'USER' };
auth.handle(userRequest); 
// 출력:
// ✅ 인증 성공
// ❌ 권한 실패: 관리자만 접근 가능합니다. (실제 주문 처리까지 못 가고 중간에 차단됨)
```

### 2. GUI 이벤트 전파 (Event Bubbling)

웹 브라우저에서 HTML 태그를 클릭했을 때 발생하는 **이벤트 버블링(Event Bubbling)** 메커니즘이 바로 이 패턴으로 구현되어 있습니다.

화면에 있는 버튼(`Button`)을 클릭하면, 버튼이 이벤트를 처리할지 결정합니다. 처리하지 않으면 버튼을 감싸고 있는 박스(`Div`), 최종적으로는 화면 전체(`Window`) 순으로 이벤트가 거슬러 올라가며 처리할 수 있는 주인을 찾습니다.

### 3. 관리자페이지 로그인

사용자가 "관리자 상품 관리 페이지"에 접속하려고 주소(URL)를 입력하면, 시스템은 뒤에서 다음과 같은 사슬을 실행합니다.

```jsx
[요청: 관리자 페이지 보여줘!] 
      ⬇️
[1번 핸들러: 인증 검사 (Authentication)]
  - "이 사람 로그인 유저 맞아?"
  - NO ❌ -> "로그인 페이지로 튕겨내고 체인 종료!"
  - YES  -> "다음 단계로 토스!"
      ⬇️
[2번 핸들러: 접근 권한 검사 (Authorization)]
  - "로그인은 했는데, '관리자' 권한이 있는 사람이야?"
  - NO ❌ -> "403 Forbidden(접근 금지) 에러 띄우고 체인 종료!"
  - YES  -> "다음 단계로 토스!"
      ⬇️
[3번 핸들러: 세부 권한별 페이지 렌더링 (Role-based Rendering)]
  - "이 관리자는 '상품 관리' 권한이 있나? 아니면 '정산 관리' 권한만 있나?"
  - "상품 관리 권한이 있군! 이 사람에겐 상품 관리 메뉴가 포함된 대시보드를 그려주자."
```

만약 이 패턴을 쓰지 않고 일반적인 코드로 작성하면, **모든 관리자 페이지(상품 관리, 회원 관리, 매출 통계 등)의 코드 첫 줄마다** 아래와 같은 똑같은 코드를 복사·붙여넣기 해야 합니다.

```jsx
// 패턴을 안 쓰면 모든 페이지 함수마다 이 짓을 해야 합니다...
function 상품관리페이지_보여주기(user) {
    if (!user.isLoggedIn) return 로그인페이지로_이동();
    if (user.role !== 'ADMIN') return 거부화면_보여주기();
    if (!user.permissions.includes('PRODUCT_MANAGE')) return 권한없음화면_보여주기();
    
    // 이제야 진짜 본론 실행...
}
```

하지만 **책임 연쇄 패턴**을 도입하면, 인증과 권한 체크라는 공통의 '문지기'들을 딱 한 번만 만들어 두고 사슬로 엮어버릴 수 있습니다. 

새로운 관리자 기능(예: `이벤트 관리 페이지`)이 추가되더라도, 실제 개발자는 권한 체크 코드를 새로 짤 필요 없이 "이 페이지 요청도 기존의 [인증 ➡️ 권한 ➡️ 세부권한] 체인을 타게 해줘"라고 설정만 해주면 끝납니다.

> 실제로 Java Spring Boot의 `Spring Security`, Node.js의 `Passport`, Next.js/React의 `Middleware/Route Guard` 같은 기술들이 전부 이 개념을 기반으로 관리자 페이지의 로그인 및 권한별 화면 노출을 제어하고 있습니다.

### 4. 한 줄짜리 연쇄 사슬 (이것도 책임 연괘 패턴일까?)

책임 연쇄 패턴의 핵심 철학인 "앞에서 처리할 수 있으면 처리하고, 없으면 다음으로 넘긴다"를 가장 미니멀하게 표현한 형태입니다.

```jsx
console.error((error.response?.data?.message || error.message));
```

여기서 `||` (OR 연산자)가 바로 **체인을 연결하는 연결고리** 역할을 하고 있습니다. 자바스크립트의 `||` 연산자는 "앞의 값이 없으면(false/null/undefined 등), 뒤의 값을 봐라"라는 규칙을 가지고 있거든요.
만약 이 한 줄짜리 사슬(`||`)을 쓰지 않았다면, 에러 하나 출력하려고 아래처럼 구구절절 `if-else` 문을 썼어야 했을 겁니다.

```jsx
// 패턴(사슬)을 쓰지 않은 복잡한 코드
if (error.response && error.response.data && error.response.data.message) {
    console.error(error.response.data.message);
} else {
    console.error(error.message);
}
```

### 🎯 참고

자바스크립트에서 이 `?.` 문법의 공식 명칭은 '옵셔널 체이닝(Optional Chaining)'입니다.
소프트웨어 공학에서는 '옵셔널 체이닝(문법)'과 '책임 연쇄 패턴(디자인 패턴)'을 기술적으로 조금 다르게 분류합니다.

### 🤝 둘이 닮은 점: "에러로 터지는 걸 막고 뒤로 넘긴다”

흐름의 철학은 완벽히 같습니다.

```jsx
error.response?.data?.message
```

이 코드도 앞에서부터 체인을 타고 가다가 **"어? `response`가 없네(null/undefined)?"** 혹은 **"어? `data`가 없네?"** 싶으면, 프로그램이 에러(Crash)를 내며 멈추는 게 아니라 **"난 모르겠다!" 하고 즉시 `undefined`를 반환하며 체인을 안전하게 종료**합니다.

"내가 처리할 수 없으면 안전하게 다음(여기서는 `undefined`라는 결과)으로 넘긴다"는 흐름 제어 측면에서는 책임 연쇄 패턴과 일맥상통합니다.

### 🔬 기술적인 차이점: "데이터의 길" vs "로직의 길"

하지만 면접이나 기술적인 대화에서는 이 둘을 명확히 구분합니다. 가장 큰 차이는 '무엇을 연결했는가'입니다.

### 1. 옵셔널 체이닝 (`?.`) ➡️ 데이터의 사슬 (Object Depth)

이것은 디자인 패턴이 아니라 언어 고유의 '문법(Syntax)'입니다.

- 겉보기에 체인처럼 보이지만, 실제로는 **거대한 데이터(객체)의 깊은 속성**으로 파고 들어갈 때 `if`문을 줄여주는 편의 기능입니다.
- 비유하자면, 서랍장(`error`) 안의 서랍(`response`) 안의 상자(`data`) 안의 편지(`message`)를 열어보는 과정입니다. 중간에 서랍이 잠겨있거나 없으면 그냥 열기를 포기하는 것입니다.

### 2. 책임 연쇄 패턴 (CoR) ➡️ 실행 로직의 사슬 (Behavior)

이것은 코드를 깔끔하게 짜는 구조적 방법론인 '디자인 패턴'입니다.

- 단순히 데이터를 조회하는 게 아니라, **"IP를 차단할 것인가?", "로그인을 시키고 다음 페이지를 보여줄 것인가?"** 같은 **실제 행동(로직)과 비즈니스 규칙**을 사슬처럼 연결한 것입니다.
- 비유하자면, 결재 서류가 과장 ➡️ 부장 ➡️ 사장 순으로 올라가며 승인을 받는 '행동의 흐름'입니다.

> **`?.` (옵셔널 체이닝):** 데이터를 안전하게 파고들기 위한 **문법적 체인**
>
> **책임 연쇄 패턴:** 실행할 로직(검증, 보안 등)을 유연하게 조립하기 위한 **설계적 체인**

### 🎯 결론

거대한 보안 프레임워크나 복잡한 클래스 구조만 디자인 패턴이 아닙니다. 작성하신 `A || B` 같은 숏코딩 안에도 "우선순위에 따라 책임을 다음으로 전가한다"는 책임 연쇄의 철학이 그대로 녹아있습니다. 개발하면서 자연스럽게 이 패턴을 이미 다 다루고 계셨던 셈입니다!

## 💡 이 패턴을 써야 하는 이유 (핵심 가치)

가장 큰 이유는 "요청을 보내는 사람(Client)"과 "요청을 해결하는 사람(Handler)"을 완벽하게 분리(Decoupling)하기 위해서입니다.

- **유연한 조립 라인:** 검사 단계나 처리 순서를 바꾸고 싶을 때, 전체 코드를 뜯어고칠 필요 없이 **사슬을 갈아끼우기만(레고 블록 조립하듯)** 하면 됩니다.
- **코드가 단정해짐:** 하나의 거대한 함수가 10가지 일을 다 하던 방식에서, 10개의 클래스가 각각 1가지만 책임지는 구조가 됩니다. (**단일 책임 원칙 만족**)
- **동적 제어 가능:** 프로그램이 실행 중(Runtime)인 상태에서도 사용자의 등급이나 상황에 따라 검사 단계를 추가하거나 뺄 수 있습니다.

## 💡 패턴을 사용하지 않았을 때의 기존 방식의 문제점

만약 이 패턴을 쓰지 않고 일반적인 `if-else`나 `switch` 문으로 처리하면 어떤 파국이 일어날까요?

❌ **괴물 같은 거대 함수 탄생 (God Object)**

하나의 함수나 클래스가 인증, 인가, 로깅, 데이터 검증, 캐싱을 다 처리해야 하므로 코드가 수천 줄로 늘어납니다.

**❌ 수정하다가 다른 기능이 고장 남 (경직성)**

새로운 검사 단계(예: "해외 IP 차단 기능")를 중간에 추가하려면, 기존의 복잡한 `if-else` 문 한가운데를 비집고 들어가 코드를 수정해야 합니다. 이때 실수로 괄호 하나 잘못 건드리면 기존에 잘 돌아가던 "로그인 인증"까지 통째로 먹통이 됩니다. (**개방-폐쇄 원칙 위반**)

**❌ 코드 중복의 지옥**

관리자 페이지용 검사 로직이 있고, 일반 회원용 검사 로직이 있을 때, 중간에 '데이터 유효성 검사'라는 공통 단계가 필요하다면? 기존 방식에서는 이 코드를 두 곳에 각각 **복사·붙여넣기** 해야 합니다.

## 💡 실무에서 개발자들이 자주 하는 실수 (⚠️ 주의점)

막상 실무에서 이 패턴을 설계하거나 필터 체인을 건드릴 때 정말 많이 하는 실수 3가지입니다.

**① 사슬 끊어먹기 (Next 호출 누락)**

가장 흔한 실수입니다. 자기가 맡은 검사를 무사히 통과했으면 `super.handle(request)`이나 `next()`를 호출해서 **다음 사슬로 바통을 넘겨야 하는데, 이걸 깜빡하고 빼먹는 경우**입니다.

- **결과:** 에러도 안 나고 조용히 프로그램이 멈춥니다. 사용자는 로그인을 잘했는데 다음 화면으로 안 넘어가고 하얀 화면만 보게 됩니다.

**② 무한 루프(Circular Chain)의 늪**

사슬을 동적으로 연결하다가 실수로 `A ➡️ B ➡️ C ➡️ A` 형태로 **꼬리가 머리를 물도록 사슬을 꼬아버리는 경우**입니다.

- **결과:** 요청이 들어왔을 때 서버가 무한 동력을 가동하며 뺑뺑이를 돌다가 `StackOverflowError`를 뱉으며 서버가 뻗어버립니다.

**③ 낙동강 오리알 요청 (엔드포인트 미지정)**

사슬의 끝(마지막 핸들러)까지 갔는데도 **그 요청을 처리해 줄 주인이 아무도 없는 상황**을 대비하지 않는 경우입니다.

- **결과:** 고객센터 전화를 돌리고 돌리다가 마지막에 "뚝-" 하고 끊기는 것과 같습니다. 요청이 증발해 버리므로, 체인의 맨 마지막에는 반드시 **"처리할 수 있는 핸들러가 없습니다"라는 예외 처리(404 에러나 기본 에러 핸들러)를 디폴트로 배치**해야 합니다.

## 모의해킹 '관리자 메뉴 URL 직접접근 인가우회 취약점 조치'

### 🔓 1. 취약점이 터졌던 원인: "사슬을 우회했다"

'관리자 메뉴 URL 직접접근 인가우회' 취약점이 존재했다는 것은, 시스템 설계가 아래와 같이 되어 있었다는 뜻입니다.

- **정상적인 흐름:** `로그인 화면` ➡️ `관리자 메인 페이지` ➡️ `비밀 메뉴 클릭` (`인증/인가 사슬`을 차례대로 통과)
- **해커의 우회 공격:** 로그인을 안 했거나 일반 유저인데도, 주소창에 `https://사이트.com/admin/delete-user` 같은 **관리자 전용 URL을 직접 입력해서 강제로 침투**함.

이게 가능했던 이유는 관리자 전용 URL을 처리하는 페이지(서버)가 요청을 받았을 때, "이 요청이 앞단의 인증/인가 사슬(체인)을 거쳐서 온 정상적인 요청인가?"를 검사하는 '문지기(핸들러)'를 두지 않았기 때문입니다. 즉, 사슬 중간을 뚝 끊고 들어온 것이죠.

### 🛡️ 2.  조치 작업의 본질

```jsx
[해커가 주소창에 URL 직접 입력: /admin/secret-page]
                       ⬇️
  [1번 문지기: AuthenticationHandler (인증 검사)]
    - "너 세션(또는 토큰) 보니까 로그인도 안 되어 있네?"
    - ❌ 여기서 체인 끊고, 로그인 페이지로 강제 리다이렉트! (접근 차단)
```

```jsx
[1번 문지기: AuthenticationHandler (인증 검사)]
    - "로그인은 되어 있네? 패스!"
                       ⬇️
  [2번 문지기: AuthorizationHandler (인가/권한 검사)]
    - "근데 너 권한이 'NORMAL_USER'잖아. 여긴 'ADMIN'만 들어와."
    - ❌ 여기서 체인 끊고, "권한이 없습니다(403 Forbidden)" 에러 출력! (접근 차단)
```

### 발생하게 된 원인 : 비대해진 공통 클래스 문제

책임 연쇄 패턴(Chain of Responsibility)과 관련 디자인 패턴들이 **실무에서 보안 취약점을 어떻게 원천 차단하는지**를 보여주는 완벽한 교과서 같은 사례

## 1. 취약점이 터졌던 진짜 이유: "느슨해진 사슬(체인) 구멍"

처음 개발할 때는 `/common/`이나 `/search/` 버킷(경로)에 "로그인 안 해도 누구나 쓸 수 있는 공통 기능(예: 주소 검색, 공개 팝업)"만 넣어두었을 것입니다. 그래서 편리함을 위해 `SessionAuthInterceptor`라는 검사 사슬(인셉터 체인)에서 이 경로들을 제외(Exclude)시켜 놓은 것이죠.

> **기존 흐름:**
>
> 일반적인 관리자 요청 ➡️ `SessionAuthInterceptor` 검사 통과 ➡️ 실행
>
> `/common/` 요청 ➡️ **검사 사슬을 아예 건너뜀(Pass)** ➡️ 실행

문제는 시간이 흐르면서 후임 개발자들이 "어? 회원 목록 팝업(`userListPopup`) 기능을 만들어야 하네? 어디다 만들지? 대충 공통 팝업이니까 `/common/` 경로에 같이 넣자!" 하고 편의상 쑤셔 넣기 시작한 것입니다.

결과적으로 **로그인 체크 사슬을 건너뛰는 통로(`/common/`)에 일급 기밀인 관리자 기능이 얹어지면서**, 해커가 주소창에 직접 URL을 치고 들어왔을 때 문지기가 아무도 검사를 안 하고 통과시켜 버리는 '인가 우회 취약점'이 발생한 것입니다.

## 2. 조치한 내용의 핵심: "단일 길목 중앙 차단 (Gateway)"

조치 내역은 책임 연쇄 패턴과 **퍼사드/게이트웨이(Gateway)** 개념을 결합해 보안 구멍을 완벽히 메운 정석적인 방법입니다.

```jsx
[모든 URL 요청] 
      ⬇️
[1단계 문지기: BasicInterceptor (단일 길목 사슬)] 
   - 예외는 없다. 무조건 여기서 1차 검문 시작!
      ⬇️
   - 🤔 "요청한 주소가 PUBLIC_URIS(화이트리스트)에 있나?"
     ├─ YES ➡️ "오케이, 로그인/로그아웃/OTP 같은건 로그인 전에도 필요하지. 통과!"
     └─ NO  ➡️ "화이트리스트에 없네? 그럼 무조건 '로그인 세션(s_usrid)' 검사한다!"
                  ⬇️
               "세션 없어? ❌ SessionException 뱉고 로그인 페이지로 튕겨내!"
```

기존에는 "특정 경로는 검사를 안 받게 해줄게"라는 블랙리스트 방식(제외 방식)이었다면,
조치 후에는 "모든 경로는 무조건 검사를 받되, 안전함이 증명된 것만 들여보내 준다"는 화이트리스트 방식(허용 방식)으로 사슬의 철학을 바꾸신 겁니다. 이렇게 하면 앞으로 개발자들이 실수로 `/common/` 에 엄격한 관리자 기능을 추가하더라도, 화이트리스트에 등록되어 있지 않기 때문에 자동으로 차단되어 안전합니다.

## 3. 실무 경험과 디자인 패턴의 결합

여기서 쓰인 `Interceptor`나 `Filter`가 기술적으로 **책임 연쇄 패턴**의 형태를 띠고 있습니다.

- `BasicInterceptor`가 요청을 받아서 자기가 정의한 규칙(화이트리스트 비교 ➡️ 세션 체크)대로 처리하고,
- 조건에 맞지 않으면 `SessionException`을 던져서 **체인을 강제로 중단**시키며,
- 조건에 맞으면 다음 체인(실제 컨트롤러 로직)으로 **요청을 토스**하기 때문입니다.

공통 메서드가 오염되면서 발생한 아키텍처의 약점을, "단일 길목(체인의 최전방 핸들러)에서 화이트리스트 기반으로 통제한다"는 논리로 해결.
